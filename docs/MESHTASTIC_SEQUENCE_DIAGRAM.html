Meshtastic Node Communication Sequences

Three scenarios: broadcast flooding, unicast via CLIENT relay, unicast via ROUTER relay

Network Topology

S1 (Broadcast)
App → N1 ↔ N2 ↔ N3 all in range of each other
S2 (Direct msg)
N1 ↔ N2 (relay only) ↔ N3 N1 and N3 out of range — N2 is the sole relay
S3 (Trace Route)
N1 ↔ N2 (fwd relay) ↔ N3 ↔ N4 (return relay) ↔ N1 asymmetric paths — forward via N2, return via N4

Step-by-Step

  1. App sends text; Node 1 builds MeshPacket with to=0xFFFFFFFF, encrypts with AES-CTR (channel PSK), sets hop_limit=hop_start=3.
  2. Node 1 broadcasts over LoRa. Both N2 and N3 are in range and receive the same frame.
  3. N2 and N3 each check isRebroadcaster() — true for CLIENT. Both schedule a relay after a long random back-off window.
  4. N2 fires first (random). It relays with hop_limit=2, relay_node=N2. N1 and N3 both receive N2's relay.
  5. Implicit ACK: Node 1 detects its own packet being rebroadcast (p->from == getNodeNum()). ReliableRouter generates an internal ACK and calls stopRetransmission().
  6. Dupe suppression: N3 receives N2's relay, recognises it as a duplicate. As a CLIENT (roleAllowsCancelingDupe=true) it cancels its own queued relay via cancelSending().
  7. Both N2 and N3 deliver the decoded text to their local app/screen.

Route Learning (Node 1)

  • trigger ACK received with relay_node=N2
  • update nodeDB[N3].next_hop = N2
  • next send NextHopRouter (no flood)
  • code NextHopRouter.cpp:104

Step-by-Step

  1. App sends DM to N3 with want_ack=true. Node 1 has no route in DB → next_hop=NO_PREF → FloodingRouter is used.
  2. Node 1 transmits. N2 receives; N3 is out of range.
  3. N2 checks isRebroadcaster() = true (CLIENT). Queues rebroadcast with long CLIENT delay window (2×CWmax×slot + rand). Also starts a retransmission timer for the relayed packet.
  4. After the long delay, N2 relays to N3 with hop_limit=2, relay_node=N2.
  5. N3 receives, decrypts (channel PSK), delivers to app. Sends ACK: Routing{error=NONE} with to=N1, request_id=0x1234.
  6. N2 receives the ACK (not for N2). NextHopRouter::sniffReceived cancels its retransmission of the original packet and relays the ACK toward N1.
  7. Route learning: Node 1 receives ACK with relay_node=N2. Updates nodeDB[N3].next_hop = N2 (last byte). Stops retransmissions.
  8. All subsequent messages to N3 use NextHopRouter directly via N2 — no flooding.

  • trigger ACK with relay_node=N2
  • update nodeDB[N3].next_hop = N2
  • 2nd message NextHopRouter direct
  • code NextHopRouter.cpp:104

Step-by-Step

  1. App sends DM to N3 with want_ack=true. No route known → FloodingRouter. Node 1 transmits.
  2. N2 (ROUTER) receives. shouldRebroadcastEarlyLikeRouter()=true → uses short priority delay (rand(0, 2×CWsize)×slot).
  3. Never cancels: roleAllowsCancelingDupe=false for ROUTER. Even if another node relays first, N2 will still transmit its copy.
  4. N2 relays to N3 quickly. N3 decrypts, delivers to app, sends ACK back.
  5. ACK flows: N3 → N2 → N1. N2 relays the ACK and stops retransmissions of the original packet.
  6. Node 1 receives ACK with relay_node=N2. Updates nodeDB[N3].next_hop=N2. Stops retransmissions.
  7. 2nd message: Node 1 finds next_hop=N2 in DB. Uses NextHopRouter with explicit next_hop field — no mesh flood needed. Latency is lower and airtime is saved.

Forward Path Fields

  • route repeated uint32 nodeNums
  • snr_towards repeated sint32 (dB * 4)
  • parallel arrays route[i] heard packet at snr_towards[i]
  • at N2 relay route=[N2] snr=[+8]
  • at N3 dest route=[N2,N3] snr=[+8,+5]

Return Path Fields

  • route_back repeated uint32 nodeNums
  • snr_back repeated sint32 (dB * 4)
  • routed via normal mesh — may differ
  • at N4 relay route_back=[N4] snr_back=[+3]
  • at N1 origin snr_back=[+3,+11]

What N1 Knows at the End

  • forward path N1-N2-N3 snr=[+8,+5]
  • return path N3-N4-N1 snr=[+3,+11]
  • link quality per-hop SNR in both directions
  • asymmetry paths differ — normal in mesh
  • back path always collected, no flag needed
  • back path code TraceRouteModule.cpp
  • proto definitionRouteDiscovery in mesh.proto

Step-by-Step

  1. App on Node 1 issues a trace route to N3. Node 1 sends TRACEROUTE_APP with to=N3, want_response=true, empty route=[] and snr_towards=[].
  2. Node 1 transmits. N2 (ROUTER) receives it at SNR = +8 dB. N3 is out of direct range.
  3. N2 appends its nodeNum and the measured SNR as parallel entries: route=[N2], snr_towards=[+8]. Relays toward N3 with short ROUTER priority delay.
  4. N3 receives from N2 at SNR = +5 dB. Appends: route=[N2,N3], snr_towards=[+8,+5]. As the destination, constructs a RESPONSE with route_back=[], snr_back=[] and sends it back to N1.
  5. The response is routed independently. N4 — a different node with better line-of-sight to N1 — receives the response from N3 at SNR = +3 dB. N4 appends to the return arrays: route_back=[N4], snr_back=[+3], and relays toward N1.
  6. N1 receives the relayed response from N4 at SNR = +11 dB. N1 appends the final SNR: snr_back=[+3,+11]. Now N1 has a full picture: forward path N1→N2→N3 with link SNRs, and return path N3→N4→N1 with link SNRs.
  7. Key insight: snr_towards[i] is the SNR at which route[i] heard the previous node — it measures the link as seen by the receiver. Asymmetric paths and SNR values are common due to terrain, antenna orientation, and transmit power differences between nodes.

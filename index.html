The Problem

What happens when cellular towers go down, the internet is unreachable, or you're simply off the grid?

Scenarios

  • Natural disasters (floods, earthquakes)
  • Remote hiking / mountaineering
  • Rural areas with no coverage
  • Events with overloaded networks
  • War zones & conflict areas

Requirements

  • No infrastructure dependency
  • Long range
  • Low power / long battery life
  • Encrypted
  • Affordable

What is Meshtastic?


  • Open-source project — 100% community-driven
  • Turns inexpensive LoRa radios (~$20-50) into a mesh network
  • Long-range, off-grid text messaging & GPS location sharing
  • No SIM card, no subscription, no internet required
  • Decentralized — every node is a router
  • End-to-end encrypted communication
  • Works with companion apps on Android, iOS, Web, and CLI

Project started in 2020 — active development with thousands of contributors

LoRa Technology

What is LoRa?

  • Long Range radio modulation technique
  • Developed by Semtech (patented chirp spread spectrum)
  • Sub-GHz ISM bands (license-free)
  • Extreme range vs. WiFi/Bluetooth
  • Very low power consumption
  • Trade-off: low data rate (~0.3 – 22 kbps)

Key Parameters

  • Spreading Factor (SF) — 7-12
    Higher = longer range, slower speed
  • Bandwidth — 125-500 kHz
    Wider = faster but less range
  • Coding Rate — 4/5 to 4/8
    Higher = more error correction
  • TX Power — up to +30 dBm
    Region-dependent limits

How Mesh Networking Works

  • Every node rebroadcasts messages it hasn't seen before
  • Hop limit decrements at each relay — prevents infinite loops
  • No dedicated routers needed — every device participates

Message Flow — Step by Step


1. Compose message on app → 2. Send via Bluetooth/WiFi → 3. Radio broadcasts (LoRa) → 4. Neighbors rebroadcast → 5. Target receives & decrypts

Managed Flooding

  • Before rebroadcasting, node listens briefly
  • If another node already rebroadcasted — skip
  • SNR-based contention: farther nodes get priority
  • Router/Repeater roles have higher priority

Next-Hop Routing (v2.6+)

  • First message uses managed flooding
  • System tracks which node relayed the ACK
  • Subsequent messages route via known path
  • Falls back to flooding if route fails

Protocol Architecture


Layer 3 Multi-Hop Messaging — Managed flooding + next-hop routing, hop limit, SNR contention
Layer 2 Reliable Zero-Hop — ACKs, retransmissions (max 3 attempts), timeout logic
Layer 1 Unreliable Zero-Hop — Single LoRa packet: 4B dest + 4B sender + 4B ID + flags + up to 237B payload
Layer 0 LoRa Physical — Chirp spread spectrum, preamble=16, sync word=0x2B, CSMA/CA

Serialization: Protocol Buffers (protobuf) — compact binary encoding for constrained links

Packet Structure



  • Headers are unencrypted — any node can relay without knowing the channel key
  • Broadcast destination: 0xFFFFFFFF
  • Flags byte includes: hop limit, want-ack bit, via-MQTT bit

Encryption & Security

Channel Messages

  • AES256-CTR symmetric encryption
  • Pre-Shared Key (PSK) per channel
  • Anyone with the key can read & send
  • Default key is public: "AQ=="
  • Up to 8 channels per device

Direct Messages (v2.5+)

  • x25519 public key cryptography
  • Each node has unique key pair
  • Encrypted with recipient's public key
  • Signed with sender's private key
  • True end-to-end encryption

Honest Limitations

  • Not as secure as TLS 1.3 or Signal
  • No Perfect Forward Secrecy (PFS)
  • Channel msgs: no sender authentication
  • Node IDs based on MAC (spoofable)
  • x25519 is not quantum-resistant
  • AES256 is quantum-resistant

Security is practical for the use case — not designed for nation-state adversaries.

Hardware Ecosystem

ESP32 / ESP32-S3

  • WiFi + Bluetooth
  • Higher power consumption
  • More RAM, lower cost
  • Good for powered/indoor nodes
  • Brands: LILYGO, Heltec

nRF52840

  • Bluetooth only
  • Very power efficient
  • UF2 bootloader (easy flash)
  • Ideal for solar / battery
  • Brands: RAK, LILYGO T-Echo

RP2040

  • Raspberry Pi chip
  • Dual-core ARM
  • Cost-effective DIY
  • Growing support
  • Brands: Raspberry Pi Pico

Radio chips: prefer Semtech SX126x / LR11xx (newer, better) over SX127x (older)

Price range: $20 – $80 for a complete node

Mesh Network Scaling

Auto-Scaling Broadcast Intervals

  • Default intervals:
    • Device telemetry: 30 min
    • Position: 15 min
    • Node info: 3 hours
  • When mesh exceeds 40 nodes, intervals auto-scale:

interval × (1.0 + (nodes - 40) × 0.075)

CSMA/CA

  • Channel Activity Detection (CAD) before TX
  • If channel busy → random backoff
  • Contention window scales with utilization

Practical Limits

  • Works best with <100 nodes per mesh
  • Message delivery is best-effort
  • Not designed for high-throughput data

Router Hierarchy

Router  (OSThread)
  └─ FloodingRouter
       recentBroadcasts dedup table
       perhapsRebroadcast()
       └─ NextHopRouter  (v2.6+)
            neighbor-aware path
            └─ ReliableRouter
                 ACK / NACK
                 3 retransmit attempts

src/mesh/Router.h · FloodingRouter.h · NextHopRouter.h · ReliableRouter.h

Packet Flow

  • Radio RX → enqueueReceivedMessage()
  • Router::runOnce() processes queue
  • Deliver to local modules via MeshService
  • Rebroadcast? → perhapsRebroadcast()
  • TX: sendLocal() → radio TX queue

Router is an OSThread — runs cooperatively via ArduinoThread scheduler

Key Data Structures

PacketHeader (12 bytes)

struct PacketHeader {
  NodeNum to;         // 4B destination
  NodeNum from;       // 4B source
  PacketId id;        // 4B unique ID
  uint8_t flags;      // hop_limit [2:0]
                      // want_ack  [3]
                      // via_mqtt  [4]
                      // hop_start [7:5]
  uint8_t channel;    // channel hint
  uint8_t next_hop;   // last byte only
  uint8_t relay_node; // last byte only
};
